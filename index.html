Step 2 - Build the checks

How the automated checks were selected

I prioritized high-risk, recurring rules that can be detected reliably from JSON and corrected before submission.

Automation boundary

Check objective evidence automatically; flag contextual or legal decisions for human review.

Prioritization framework

Four selection criteria

01Impact

Likely to cause rejection or block processing.

02Evidence

Found in rejected samples or official guidance.

03Confidence

Detectable reliably from the submitted JSON.

04Fixability

Supports a specific field-level correction.

Selected for automation

Checks implemented in the validator

Deterministic
Machine decision → Error

Clear structural or formatting condition; the same JSON always produces the same result.

Hybrid
Risk signal → Warning

Uses contextual signals for human confirmation; never decides approval or rejection automatically.

P0Deterministic

Payload and core structure

Readable JSON/ZBS pseudo-JSON with root, sections[], and user-visible content.

Why selected
Nothing else can be validated or rendered when the payload is malformed. Detection is exact and the fix is immediate.
Validator output
Returns an Error for unreadable input, missing root/sections, or missing visible content.
P0Deterministic

Parameter syntax and rendering

Variables must use a complete <parameter_name> pattern with supported characters.

Why selected
Parameter defects appear in rejected samples, are detectable directly from text fields, and can be fixed at the exact location.
Validator output
Reports malformed or exposed parameters and provides a corrected parameter example.
P0Hybrid

Customer identification

Look for a clearly labelled customer name, customer code, phone suffix, or equivalent recipient identifier.

Why selected
Recipient context is central to a business notification and is a recurring reject reason in the sample set.
Detection limit
The tool can verify presence and labelling, but cannot prove that the runtime value belongs to the correct person.
P0Hybrid

Transaction or business-context identification

Look for a labelled order, contract, appointment, account, invoice, or other event identifier.

Why selected
It distinguishes a specific service notification from generic outreach and directly addresses provided rejection evidence.
Detection limit
Keyword and parameter evidence can show likely context, but semantic relevance still requires moderation judgment.
P0Deterministic

Phone numbers and links placed in CTA

Detect phone numbers or navigable URLs in message body content instead of an action button.

Why selected
The provided samples contain this explicit reject reason. Regex detection is reliable and the remediation is unambiguous.
Validator output
Highlights the exposed value and recommends moving it to a phone or URL CTA.
P1Deterministic

CTA action and destination integrity

Buttons need a supported action and valid destination data, such as a URL or phoneCode.

Why selected
A visible but broken CTA harms both submission quality and the end-user journey. Required fields are structurally testable.
Validator output
Identifies missing or incompatible action data and names the field that must be supplied.
P1Deterministic

Unsafe or unsupported destinations

Flag malformed URLs and destinations associated with personal chats, groups, or unsupported redirects.

Why selected
Destination strings are available in JSON and unsupported routing appears in the rejected samples.
Detection limit
The tool validates patterns and known destinations; it cannot guarantee the safety or ownership of every external page.
P1Deterministic

Template-type component requirements

Payment, Voucher, Rating, and other types must include their corresponding utility components.

Why selected
A type-to-component matrix is consistent, machine-readable, and catches failures before the payload reaches moderation.
Validator output
Names the missing component, such as open_utility or rating, based on the selected type.
P1Hybrid

Parameter labels and prefixes

Dynamic values should be paired with labels that explain what each value represents.

Why selected
Unlabelled variables are documented in the reject samples and reduce message clarity for both moderators and users.
Detection limit
Map keys provide strong evidence, while free-text labels require a heuristic and should be presented as a warning.
P1Hybrid

Basic content-quality risks

Flag known rejection typos, excessive all-caps copy, and repeated punctuation or characters.

Why selected
These checks prevent avoidable review cycles and demonstrate actionable pre-submission guidance.
Detection limit
Only high-confidence patterns should trigger issues; broader language quality remains a recommendation, not an approval decision.

Automation boundary

Mapped, but not used for automatic rejection

Legal and licensing

Industry licences, promotion legality, and regulatory eligibility require external evidence unavailable in template JSON.

Business ownership

Bank-account, brand, image, and destination ownership must be confirmed against business records.

Visual and semantic meaning

Misleading imagery, sensitive meaning, and full Tag suitability depend on context and human interpretation.

Historical user relationship

Consent, past transactions, send frequency, complaints, and recipient correctness require CRM or delivery-history data.

Prototype policy: these rules may create a manual-review flag or risk signal, but never an automatic claim that Zalo will reject or approve the template.

Tool roles

Why I used more than one AI tool

Claude Thinking and content partner

Used for framing the product problem, comparing solution directions, and pressure-testing the explanation of moderation rules.

Codex Code implementation partner

Used to read the codebase, implement UI and validation changes, debug behavior, and adjust the prototype based on review feedback.

Perplexity Source discovery partner

Used to find relevant public sources quickly, then checked manually against the official Zalo rule source before using them.

Manual review Final quality gate

Used to verify wording, product logic, rule interpretation, and whether each automated check should be an error, warning, or manual flag.

Prompting method

Two-layer prompting to reduce wrong outputs

Layer 1 Context interviewer

The first AI layer is used to understand the assignment context, ask back unclear points, and turn the clarified problem into a precise execution prompt.

Layer 2 Task executor

The second AI layer receives that refined prompt and performs the task, such as coding, restructuring content, or checking implementation details.

Quality control

How I avoided AI loops and hallucinations

Switch models when stuck

I switched between Claude and Codex when the discussion became repetitive or too attached to one implementation path.

Check official sources by hand

Perplexity helped with discovery, but I manually checked the official Zalo source before turning a claim into a rule.

Keep scope labels clear

Objective JSON checks become errors; contextual signals become warnings; external evidence stays manual.

Use AI edits as drafts

Codex implementation was reviewed and adjusted according to the product goal, UI clarity, and assignment requirements.

Final ownership

The final prototype is not an AI-generated answer pasted into a submission. AI helped me move faster, but I chose the product scope, selected the automation boundaries, reviewed the rule map, and edited the interface based on the assignment goals.
